# WHETSTONE — End-to-End Training Procedure

A step-by-step operational procedure for installing and preserving a target reasoning style (here: compact symbolic reasoning) through verifiable-reward RL, with no external teacher of higher capability.

This document is the implementation procedure for the recipe described in the WHETSTONE paper (`whetstone_blog.md`). The paper defines *what* the recipe does and *why*; this document defines *how* an AI agent runs it end-to-end, including every command, gate, checkpoint, and recovery path. The procedure is hardware-agnostic and model-agnostic: it runs on any GPU configuration that can host the chosen base model in BF16 with vLLM for inference and FSDP-style sharded training for SFT/RL, and on any base LLM that produces parseable `<think>` blocks under the Step 2.1 calibration probe. Section 1 lists the knobs to scale across GPU configurations.

The five stages of the recipe correspond one-to-one with five executable stages below.

---

## 1. Preconditions

### 1.1 Conventions and placeholders

The same base model plays three roles throughout — harvest generator (Stage 1), compressor (Stage 2), and student (Stages 3–5) — and all training data is generated by that base model itself. The cross-family auditor used at Stages 2.5 and 4 acts only as a filter; it never supplies solutions or operates at inference time. Stage labels "v4b-spike" and "audit-passed" refer to the Stage 3 and Stage 4 SFT checkpoints respectively. "The verifier" is the deterministic answer comparator (`cavethought.verify.verify_response` in the reference implementation, or any equivalent).

Commands below use these placeholders. Substitute with values from your environment before launching anything.

| Placeholder | Meaning | Example |
|---|---|---|
| `<DATA_ROOT>` | Persistent root for all run artifacts (data, checkpoints, logs) | `/data/whetstone` |
| `<MODELS_ROOT>` | Persistent root holding base-model weight directories | `/data/models` |
| `<BASE_MODEL>` | Directory name of the base LLM under `<MODELS_ROOT>` | `<your-base-model>` |
| `<RUN>` | Stable identifier for this pipeline run | `<base>_<pool>_v1` |
| `<POOL>` | Input problem pool stem | `math_dapo_train` |
| `<N_GPUS>` | GPUs available on the training node | `8` |
| `<TP>` | vLLM tensor-parallel size (≤ `<N_GPUS>`) | `8` |
| `<NODE>` / `<NODE_A>`, `<NODE_B>` | Hostnames or local for the train and (optional) second node | `node-a` |
| `<PYTHON>` | Python interpreter with the dependency stack installed | `python3` or a venv path |
| `<SCRATCH>` | Ephemeral scratchpad (e.g. `/tmp`) — never write load-bearing artifacts here | `/tmp` |

### 1.2 Hardware

The procedure runs on any GPU node that can hold the base model in BF16. The validated reference is 8×80GB GPUs, but the only hard requirement is "enough aggregate VRAM to load the base model + KV cache for the chosen context length." Scaling notes:

- **vLLM (inference, used in Stages 1, 4, 5.0).** Set `tensor_parallel_size = <TP>` to the number of GPUs you want to dedicate per vLLM instance. If `<TP>×per_GPU_VRAM` is just enough to host the model + activations, lower `gpu_memory_utilization` (default 0.85–0.90) until allocation succeeds, or split into N workers each with `tp=1` running on disjoint pool slices.
- **FSDP training (Stages 3, 4 SFT, 5 DAPO).** Effective batch size in the validated config is 64 = `per_device_train_batch_size=1 × gradient_accumulation_steps=8 × 8 GPUs`. To preserve effective batch size on different GPU counts, scale `gradient_accumulation_steps` inversely: with 4 GPUs use `grad_accum=16`; with 16 GPUs use `grad_accum=4`. Do not change `per_device_train_batch_size` unless VRAM forces you to — increasing it changes the surprisal-weighted gradient noise floor.
- **Smaller-VRAM GPUs (e.g. 40GB).** For models above ~9B you will need to either (a) raise `<TP>`, (b) shorten `max_model_len` and `max_tokens`, or (c) drop to a smaller base model. Surprisal weighting and the rest of the recipe are unchanged.
- **Multi-node.** Stage 1 harvest and Stage 5.0 curriculum K=8 trivially parallelize across N nodes by splitting the problem pool into N disjoint slices (e.g. by `_uid` hash mod N). All other stages run on a single node.

### 1.3 Storage

- **Persistent root `<DATA_ROOT>`.** Mount a filesystem that survives node restarts and container resets. Every load-bearing artifact (raw data, compressed traces, audit verdicts, SFT/DAPO checkpoints, logs) lives under `<DATA_ROOT>/` so the resume-safe scripts can pick up where they left off. See §10 for the directory layout.
- **Free space, order of magnitude.** ~2× base-model-weights bytes for storing the base model + each merged SFT/DAPO checkpoint, plus pool-size-dependent space for harvest JSONL (rough rule: K × pool_size × ~20 KB per rollout for math-tier pools at the default 32k token budget). End-to-end footprint for a typical run (K=4 harvest + K=8 regeneration + 2 SFT checkpoints + 2 DAPO checkpoints) is dominated by the SFT/DAPO checkpoints; budget at least 2× the merged-checkpoint bytes per saved step in `<DATA_ROOT>/sft_runs/` and `<DATA_ROOT>/dapo_runs/`.
- **Ephemeral scratch `<SCRATCH>`.** Use only for write-throwaway helpers (`harvest.py` payloads, temporary scripts). Never store outputs, logs, or checkpoints here. If `<SCRATCH>` is wiped on container reset (the common case), anything written there is lost on restart.

### 1.4 Other preconditions

Verify all of the following before starting. Skipping any one will cause failure later at significantly higher cost.

1. **Dependency stack.** A Python environment with Torch ≥ 2.10 (CUDA-matched), vLLM 0.19.1, flash-attn ≥ 2.8.3, FLA, causal-conv1d, and (when running vLLM v1 on certain GPU configurations) the Triton patch. If your environment lives on a non-persistent volume, document a one-shot bootstrap script and re-run it after any node reset.
2. **Base model on disk.** Confirmed at `<MODELS_ROOT>/<BASE_MODEL>`. The base must *not* already be an instruct-tuned thinking model with suppressed `<think>` blocks — instruct-tuned thinking checkpoints have already collapsed the high-entropy decision-token distribution that Stage 1 depends on. Verify with the Step 2.1 calibration probe before committing GPU-days to harvest.
3. **Problem pool.** JSONL at `<DATA_ROOT>/data/interim/<POOL>.jsonl` with one record per problem and required fields: `_uid` (str), `prompt` (str), `ground_truth` (str). Optional but recommended: `level` (a difficulty bucket), `subject`. Golds must be reliable — the entire pipeline depends on deterministic verification at every stage.
4. **Cloud API key for the cross-family auditor.** `CLOUD_API_KEY` exported on the node that will run Stages 2.5 and 4 audits. Map it to whatever environment variable the chosen auditor's SDK reads (e.g. the SDK's own key variable). If you swap the auditor for a different cross-family model, replace the `--model` flag in §4 and §6 accordingly.
5. **Compute time.** Stage 1 harvest dominates total wall-clock; expect on the order of several GPU-days on the 8×80GB reference, scaling roughly inversely with aggregate FLOPS for other hardware.

Do **not** start the pipeline if your base model already produces compact reasoning, if your problem corpus lacks reliable golds, or if your target distribution is frontier-hard for your base (base 8-sample pass rate below ~0.10) — see the compactness-rigidity tradeoff in §12.

## 2. Stage 1 — Blind Harvest

**Purpose.** Sample verified-correct reasoning traces from the base model given only the problem statement, with no gold-answer conditioning. Blindness is non-negotiable: traces generated with gold visible are rationalizations, not reasoning, and lack the high-entropy decision-token distribution that downstream stages depend on. Formally, build the harvest set

```
H = { (q, trace, a*) : verifier(answer(trace), a*(q)) = 1 }
```

over rollouts drawn from the base model with the problem statement alone.

### Step 2.1 — Prompt calibration probe (mandatory, do this first)

Mass harvest costs GPU-days. A 30-minute calibration probe on a 50-problem stratified slice catches problems that would otherwise waste the full run. Do not skip.

Build a stratified 50-problem slice:

```python
import json, random
from collections import defaultdict
random.seed(0)
by_bucket = defaultdict(list)
with open("<POOL>.jsonl") as f:
    for line in f:
        r = json.loads(line)
        key = (r.get("subject", "_"), r.get("level", "_"))
        by_bucket[key].append(r)
sample = []
per_bucket = max(1, 50 // len(by_bucket))
for k, rows in by_bucket.items():
    sample.extend(random.sample(rows, min(per_bucket, len(rows))))
sample = sample[:50]
with open("<POOL>_calib50.jsonl", "w") as f:
    for r in sample:
        f.write(json.dumps(r) + "\n")
```

Run the Step 2.2 harvest script on `<POOL>_calib50.jsonl` with the exact `K`, `temperature`, and `max_tokens` you plan to use on the full pool. Then inspect these five metrics:

1. **Format compliance.** ≥80% of rollouts produce a parseable `<think>` block followed by a final answer. If lower, the system prompt or chat template is wrong for this model — fix before mass launch.
2. **Verifier acceptance shape.** Hand-spot-check 10 verifier decisions. A verifier that rejects `"10,000,000"` against gold `"10000000"` silently halves your yield.
3. **Per-level pass-rate distribution.** Roughly U-shaped (easy ~90%, hard ~10%) is healthy. Uniformly low across all levels means prompt/sampling config is broken, not pool difficulty.
4. **Median think tokens per level.** Must grow with level. If flat (~500 tokens median everywhere), the model is truncating rather than reasoning — raise `max_tokens` or lower `temperature`.
5. **Token-cap-hit rate.** <10% of rollouts should hit `max_tokens` without `</think>`. If higher, raise `max_tokens` or drop the hardest level from the pool.

**Decision rule.** If any of the five metrics fails, fix the prompt or config, then re-run the 50-problem probe. Do **not** proceed to mass harvest until all five are healthy. Probe cost ≈ 30 GPU-minutes; wasted full harvest cost ≈ 3 GPU-days.

### Step 2.2 — Sampling configuration

Validated harvest configuration (tested across multiple base-model families in the 9B–27B range):

| Parameter | Value | Notes |
|---|---|---|
| `K` | 4 (or 8) | K=8 trades GPU-hours for coverage |
| `temperature` | 0.6–0.9 | 0.9 on harder pools widens exploration |
| `top_p` | 0.95 | |
| `max_tokens` | 32,000 | Default harvest budget. Raise to 40k+ only for frontier-difficulty pools when Step 2.1 shows >25% cap-hit at 32k. Going below 32k truncates legitimate hard-tier reasoning |
| `max_model_len` | `max_tokens + 1024` | 33,024 at the 32k default; 1k prompt headroom |
| vLLM `tensor_parallel_size` | 8 | Or per-GPU TP=1 with N workers per machine |
| `gpu_memory_utilization` | 0.85–0.90 | |

### Step 2.3 — Write the harvest script

Create `<SCRATCH>/harvest.py` on the target node:

```python
import argparse, json, os, sys

def main():
    sys.path.insert(0, "<DATA_ROOT>")  # so `cavethought.verify` resolves
    from vllm import LLM, SamplingParams

    ap = argparse.ArgumentParser()
    ap.add_argument("--input", required=True)
    ap.add_argument("--output", required=True)
    ap.add_argument("--model", required=True)
    ap.add_argument("--K", type=int, default=4)
    ap.add_argument("--temperature", type=float, default=0.9)
    ap.add_argument("--top_p", type=float, default=0.95)
    ap.add_argument("--max_tokens", type=int, default=32000)
    ap.add_argument("--max_model_len", type=int, default=33024)
    ap.add_argument("--tp", type=int, default=8)
    ap.add_argument("--gpu_mem", type=float, default=0.90)
    args = ap.parse_args()

    # Resume by (uid, candidate_idx) — never by global line count.
    seen = set()
    if os.path.exists(args.output):
        with open(args.output) as f:
            for line in f:
                try:
                    r = json.loads(line)
                    seen.add((r["_uid"], r.get("candidate_idx", -1)))
                except json.JSONDecodeError:
                    pass
        print(f"[resume] {len(seen)} pairs done", flush=True)

    problems = []
    with open(args.input) as f:
        for line in f:
            r = json.loads(line)
            uid = r["_uid"]
            prompt = r.get("prompt") or r.get("problem") or ""
            gold = r.get("ground_truth") or r.get("gold") or ""
            for k in range(args.K):
                if (uid, k) in seen:
                    continue
                problems.append({"uid": uid, "k": k, "prompt": prompt, "gold": gold})
    if not problems:
        print("done"); return
    print(f"[load] {len(problems)} to generate", flush=True)

    PROMPT = (
        "<|im_start|>system\nPlace all your step-by-step reasoning between <think> and </think> tags. "
        "After </think>, give the final answer.<|im_end|>\n"
        "<|im_start|>user\n{user}<|im_end|>\n<|im_start|>assistant\n<think>\n"
    )
    llm = LLM(model=args.model, dtype="bfloat16", tensor_parallel_size=args.tp,
              gpu_memory_utilization=args.gpu_mem, max_model_len=args.max_model_len,
              trust_remote_code=True, enforce_eager=False)
    sp = SamplingParams(temperature=args.temperature, top_p=args.top_p,
                        max_tokens=args.max_tokens, seed=None)

    BATCH = 32
    out_f = open(args.output, "a")
    for i in range(0, len(problems), BATCH):
        chunk = problems[i:i+BATCH]
        outs = llm.generate([PROMPT.format(user=p["prompt"]) for p in chunk], sp)
        for p, out in zip(chunk, outs):
            rec = {
                "_uid": p["uid"], "candidate_idx": p["k"], "prompt": p["prompt"],
                "ground_truth": p["gold"], "completion": out.outputs[0].text,
                "n_tokens": len(out.outputs[0].token_ids),
                "model": args.model.split("/")[-1],
                "temperature": args.temperature, "top_p": args.top_p,
            }
            out_f.write(json.dumps(rec) + "\n")
        out_f.flush()
        print(f"[gen] {i+len(chunk)}/{len(problems)}", flush=True)

if __name__ == "__main__":
    main()
```

### Step 2.4 — Launch harvest

Launch detached so it survives SSH disconnect. Output must live under `<DATA_ROOT>/` (persistent), never under `<SCRATCH>/` (ephemeral).

```bash
nohup bash -c 'CUDA_VISIBLE_DEVICES=<GPU_LIST> \
  <PYTHON> <SCRATCH>/harvest.py \
    --input  <DATA_ROOT>/data/interim/<POOL>.jsonl \
    --output <DATA_ROOT>/data/raw/<RUN>_harvest_k4_T09.jsonl \
    --model  <MODELS_ROOT>/<BASE_MODEL>' \
  > <DATA_ROOT>/logs/<RUN>_harvest.log 2>&1 </dev/null &
disown
```

If running on two nodes in parallel, split the pool into disjoint slices by `_uid` hash mod 2 before launching. Two workers writing to the same output file corrupt the JSONL.

### Step 2.5 — Checkpointing and resume invariants

Harvest is the longest single stage (1–4 GPU-days). Containers reset without warning. Preserve these invariants in any adaptation:

1. **Append-only output**, opened in `"a"` mode. Each `(uid, k)` is flushed immediately after its batch completes. Survives SIGKILL and container reset.
2. **Per-batch fsync cadence.** With `BATCH=32`, durable progress is ~every 30 seconds of wallclock. Acceptable loss on crash ≤ 32 rollouts.
3. **Resume scan on startup.** Build `seen = {(uid, k), …}` from the existing output file and skip every pair already present. Tolerate a JSON-decode error on the *last* line (in-progress write at crash time) — do not reject the whole file.
4. **`<DATA_ROOT>` only.** Output path and log file must both live under `<DATA_ROOT>/`. `<SCRATCH>` (e.g. `/tmp`) wipes on container reset.
5. **Disjoint pool splits across machines.**
6. **Per-worker output files when TP=1 multi-worker.** With N workers per machine, each writes to `<RUN>_harvest_workerN.jsonl`. Concatenate at end. Never have N workers append to a shared file.

### Step 2.6 — Progress monitoring

```bash
ssh <host> "wc -l <DATA_ROOT>/data/raw/<RUN>_harvest_worker*.jsonl; \
            tail -3 <DATA_ROOT>/logs/<RUN>_harvest_worker*.log"
```

### Step 2.7 — Recovery from node or container reset

1. If the Python environment lives on a non-persistent volume, re-run the bootstrap from §1.4 to restore it. Anything written to `<SCRATCH>` (including `<SCRATCH>/harvest.py`) is gone; re-create the helper script.
2. Re-launch the harvest with the *same command and same output path*. The resume scan picks up where it left off. Do not start fresh and do not rename the output file.

### Step 2.8 — Verification gate (deterministic correctness filter)

After harvest completes, filter by the deterministic verifier. Keep only `(uid, k)` pairs whose final answer matches gold under `cavethought.verify.verify_response`:

```python
from cavethought.verify import verify_response
import json
with open("<RUN>_harvest_k4_T09.jsonl") as f_in, \
     open("<RUN>_harvest_verified.jsonl", "w") as f_out:
    for line in f_in:
        r = json.loads(line)
        if verify_response(r["completion"], r["ground_truth"]):
            f_out.write(line)
```

**Expected yield.** ~30–60% verified on mid-difficulty pools (base 8-sample pass rate ~0.5); ~10–20% on competition-difficulty pools (base 8-sample pass rate ~0.2–0.3). If your yield is far outside this band, recheck the Step 2.1 calibration metrics before scaling up.

### Step 2.9 — Stage 1 known failure modes

| Symptom | Cause | Fix |
|---|---|---|
| vLLM `AssertionError: Initial free memory < current free memory` | v1 memory-profile race on first launch | Retry 2–3× with fresh GPU state. If persistent, soft-patch `vllm/v1/worker/gpu_worker.py:410` (assert → warning + clamp). Requires explicit user authorization. |
| >50% rollouts hit `max_tokens` | Pool too hard for this base model | Raise `max_tokens` or filter the hardest level out. |
| Corrupt JSONL lines mid-file | Two workers writing to the same output | Use per-worker output paths; merge after run completes. |

**Output of Stage 1.** `<RUN>_harvest_verified.jsonl`. Proceed to Stage 2.

## 3. Stage 2 — Chunkwise Self-Compression

**Purpose.** Rewrite each verified-correct trace into the compact register — terse symbolic notation, numbered micro-steps, semicolon-separated claims, minimal connective prose. The compressor is the **same base model** that produced the harvest (central-model principle: no external teacher).

### Step 3.1 — Chunkwise prefill algorithm

For each problem, with original thinking split into N chunks:

1. **Chunk.** Split `thinking` into ≤`MAX_CHUNKS` paragraphs subject to a per-chunk token cap (default cap: 16 chunks).
2. **At depth k = 1..N**, build one prompt:
   - **System:** compression instructions (preserve facts, terse notation, one to a few lines per chunk).
   - **User:** `PROBLEM: {prompt}` then `ORIGINAL CHUNK 1: …` through `ORIGINAL CHUNK k: …` (all originals up to current depth).
   - **Assistant prefill:** `COMPACT CHUNK 1: <prev>` … `COMPACT CHUNK k-1: <prev>`, then `COMPACT CHUNK k: ` ← model completes only chunk k.
3. **Stop tokens** `\nCOMPACT CHUNK`, `\nORIGINAL CHUNK`, `<|im_end|>` prevent auto-continuation into chunk k+1.
4. **Depth-batch across problems.** At depth k, vLLM batches one prompt per problem with ≥k chunks. 1,000 problems × median 8 chunks ≈ 8 vLLM rounds — this is the throughput win.
5. **Concatenate** `compacts_per_chunk` with `; ` → final compact trace.

The four design choices: causal accumulation (each compact chunk conditioned on prior compacts), originals stay visible (model can decide which prior facts remain load-bearing), register anchoring (prior compacts in the prefill keep the model in-style), and failure localization (a bad compact concentrates in one chunk slot).

### Step 3.2 — Launch

```bash
ssh <NODE_A> '<PYTHON> \
  <DATA_ROOT>/scripts/compress_local_versionB.py \
    --input  <DATA_ROOT>/data/raw/<RUN>_harvest_verified.jsonl \
    --output <DATA_ROOT>/data/processed/<RUN>_compactB.jsonl \
    --model  <MODELS_ROOT>/<BASE_MODEL> \
    --tp 8 --gpu-mem 0.85 --max-model-len 65536 \
    --max-tokens-per-compact 256 --max-chunks 16 \
    --temperature 0.0 --top-p 1.0'
```

### Step 3.3 — Config knobs

| Knob | Default | Effect |
|---|---|---|
| `--max-chunks` | 16 | Hard cap on chunks per problem. Long traces get merged into the cap. Raising above ~24 hurts per-chunk compression quality (model loses track of running compact). |
| `--max-tokens-per-compact` | 256 | Per-chunk budget. 256 fits 1–3 line compacts; lower further loses load-bearing facts. |
| `--temperature` | 0.0 | Greedy. Do **not** sample — chunkwise sampling is high-variance and the downstream Δlogp gate would reject the noise anyway. |
| `--first-candidate-only` | off | Dedupe to one rollout per `_uid` when Stage 1 kept all K candidates. |
| `--max-model-len` | 65536 | Must hold (problem + all originals + all compacts + 256 generation). Drop problems where the prompt would exceed `max_model_len − max_tokens_per_compact − 32`. |

### Step 3.4 — Implementation invariants

Production reference: `<DATA_ROOT>/scripts/compress_local_versionB.py`. Preserve these in any adaptation:

- Paragraph splitter `chunk_thinking` with `MAX_CHUNK_TOKENS` per-chunk cap and `merge_and_cap` enforcing `max_chunks`.
- Prefill prompt assembly via `apply_chat_template(..., enable_thinking=False)` followed by manual append of the partial assistant turn.
- `clean_compact_lines` strips trailing `\nCOMPACT CHUNK N+1` / `\nORIGINAL CHUNK N+1` past the stop, joins multi-line compacts with `| `.
- Periodic checkpoint every 3 depths to `--output`, writing only completed problems.

### Step 3.5 — Output schema

```json
{
  "_uid": "...",
  "src_candidate_idx": 0,
  "compress_idx": 0,
  "prompt": "...",
  "ground_truth": "...",
  "thinking_original": "...",
  "thinking_original_tokens": 4823,
  "n_chunks": 8,
  "chunks": ["...", "..."],
  "compacts_per_chunk": ["...", "..."],
  "compact": "compact1; compact2; ...; compactN",
  "compact_tokens": 412,
  "compression_ratio": 0.085
}
```

**Expected distribution** for self-compression of a base model's own correct rollouts (measured on a mid-size base, ~9B): median compression ratio ≈ 0.09 (≈11×), p10 ≈ 0.03, p90 ≈ 0.25. Ratios above 0.5 typically mean the original was already short.

### Step 3.6 — Sufficiency gate (Δlogp filter)

Run *after* Stage 2 and *before* Stage 3 SFT. Reject compressions that drop too much information by checking whether the compressed trace improves the base model's conditional likelihood of the gold answer:

```
delta = log P_base(a* | q, compact_trace) − log P_base(a* | q)
keep iff delta > 0
```

Implementation: `scripts/perplexity_score.py`. **Expected pass rate ≈ 70%.** The ~30% that fail concentrate on problems with critical case-splits or verification steps that the compact register tends to elide. Reject rather than keep — they install sloppy reasoning into the policy. Gate on Stage 3 entry, not inside Stage 2, so the raw compression artifacts remain available for diagnosis.

### Step 3.7 — Inviolable invariant

Only compress traces that are **already verified correct** by the Stage 1 deterministic verifier. Compressing incorrect traces installs wrong reasoning into SFT data; the downstream policy will be subtly broken in ways that do not surface until the RL endpoint eval. Non-negotiable.

**Output of Stage 2.** `<RUN>_compactB.jsonl`. Proceed to Stage 2.5.

## 4. Stage 2.5 — Compression Process Audit

**Purpose.** The Δlogp gate from §3.6 catches *answer-recoverability* loss but is blind to several quality failures the SFT trainer cares about: hallucinated steps not in the original, dropped self-correction moments, verbose-prose creep, caveman-style degeneration, and "compactly-restated-the-problem" no-ops that recover the answer trivially. Stage 2.5 catches these process failures via a cross-family LLM auditor.

### Step 4.1 — Auditor choice

Use a model from a **different family** than the compressor (the compressor is the same base model as the student, so they share blind spots). Default: **Claude Sonnet 4.6 / Minimax-M3** via cloud API. Same auditor as Stage 4 — reusing it across stages keeps the audit lens consistent.

**Throughput.** Each audit consumes ~3k input tokens (VERBOSE + COMPACT both shown) and ~200 output tokens. At concurrency 25, on the order of 6,000+ audits complete in ~5–10 minutes wall-clock.

### Step 4.2 — Audit schema and verdict

Auditor returns strict JSON with these criteria:

1. `FAITHFUL` — every claim, equation, intermediate result in COMPACT appears (in some form) in VERBOSE ORIGINAL. No hallucinated content.
2. `PRESERVES_LOAD_BEARING` — a reader given only PROBLEM + COMPACT could retrace to GOLD without inventing steps. All required variables, constants, decompositions, derivation path present.
3. `COMPACT_REGISTER` — terse symbolic notation (`=`, `⇒`, `→`, `✓`, `⚠`, `?`). NOT verbose prose. NOT caveman ("compute. add. done."). NOT a copy of the verbose original.
4. `PRESERVES_SELF_CORRECTION` — if VERBOSE had "wait", "actually", "no, this gives…", the COMPACT preserves them with `⚠` / `?` markers or explicit retraction. `null` if N/A.
5. `COMPRESSION_QUALITY` — integer 1 (broken) … 5 (excellent), rubric in `audit_compressions.py`.

**Verdict.** `pass` iff `faithful AND preserves_load_bearing AND compact_register AND quality >= 3`, else `fail`.

### Step 4.3 — Calibration (do this first)

Run the auditor on a **stratified 50–100 sample** spanning all difficulty levels. Hand-spot-check 10 verdicts. If the auditor over-rejects (many `fail` that look reasonable on inspection) or over-accepts (many `pass` with visible compression bugs), tune the system prompt before bulk. Bulk-running with a broken prompt contaminates the Stage 3 SFT corpus, so calibration up-front is mandatory.

### Step 4.4 — Bulk audit launch

```bash
CLOUD_API_KEY=<key> <PYTHON> \
  <DATA_ROOT>/scripts/audit_compressions.py \
    --input   <DATA_ROOT>/data/processed/<RUN>_compactB.jsonl \
    --output  <DATA_ROOT>/data/processed/<RUN>_compactB_audited.jsonl \
    --concurrency 25 \
    --model claude-sonnet-4-6
```

Resume-safe: append-only output, skips already-audited `(uid, src_candidate_idx)` on restart.

### Step 4.5 — Filter to audit-passed corpus

Join compactB with audit verdicts on `(uid, src_candidate_idx)` and keep only `verdict == "pass"` rows. **Expected pass rate 70–90%** on a healthy compressor. If <60%, the compression prompt or chunk-size is off — iterate Stage 2 before proceeding.

### Step 4.6 — Local subagent fallback

If no cloud API access, fan out N subagents (via the Agent tool), each reviewing ~200 compressions with the same JSON-output prompt. This does not scale past ~1k audits due to subagent context limits; for larger pools use the cloud API.

## 5. Stage 3 — Compressibility-Aware Cold-Start SFT

**Purpose.** Install the compact register as a stable mode of the policy. The trick is **compressibility-aware (surprisal) weighting**: reweight the SFT loss by the base model's token-level surprisal so the gradient concentrates on novel reasoning content (~20% of tokens) rather than fluency boilerplate (~80%).

Formally, with base-model per-token surprisal `S_base(x_t | x_<t) = −log P_base(x_t | x_<t)`:

```
w_t = max(1, 1 + alpha · S_base(x_t | x_<t)),    alpha = 0.5
L_SFT = − E_trace  Σ_t  w_t · log P_policy(x_t | x_<t)
```

High surprisal signals low-compressibility reasoning moves; the Stage-2 likelihood gate (§3.6) and the Stage-4 audit (§6) catch unsupported compact jumps from the other side, so this inversion is safe.

### Step 5.1 — Inputs

- `<RUN>_compactB_audited.jsonl` filtered to audit-pass (§4.5).
- Base model at `<MODELS_ROOT>/<BASE_MODEL>`.

### Step 5.2 — SFT config (validated; tune learning rate and batch size to your base model and GPU count)

|---|---|---|
| Parallelism | Full-parameter FSDP2 | **Not** LoRA / QLoRA / DeepSpeed |
| Precision | BF16 mixed | Gradient checkpointing on |
| Optimizer | AdamW | |
| Learning rate | **2e-5 for base models** | `2e-4` is the Instruct recipe — using it on a base destroys think-format learning |
| LR schedule | Cosine | |
| Epochs | 1 | Over the audit-passed corpus |
| Effective batch size | 64 | `per_device=1`, `grad_accum=8`, 8 GPUs |
| Surprisal weight alpha | **0.5** | In `w_t = max(1, 1 + alpha · S_base)` |
| Loss masking | Assistant-only | Do not train on system/user tokens |

## 7. Stage 5 — Style-Preserving DAPO

**Purpose.** Reinforcement-learn from the audit-passed SFT checkpoint while preserving the compact register through three reinforcing pressures: a small KL anchor to the styled SFT reference, within-group DAPO length ranking, and a soft length penalty in the format reward. Formally, with `policy` being the trainee and `sft_ref` being the audit-passed checkpoint:

```
L_DAPO = − E[ min(rho · A, clip(rho, 1−eps_low, 1+eps_high) · A) ]  +  beta · KL(policy || sft_ref)
```

with beta=0.005, eps_low=0.2, eps_high=0.28. The small but nonzero beta is the WHETSTONE deviation from vanilla DAPO — it makes the KL a **load-bearing stylistic anchor**, not a generic regularizer.

### Step 7.1 — Curriculum K=8 inference (REQUIRED before DAPO launch)

The pass-rate distribution used to build the curriculum must come from running K=8 on train **with the exact model that will initialize DAPO** (the audit-passed checkpoint). Reusing a Stage-3 or Stage-4a K=8 run is wrong: different models have different capability boundaries, and a problem that's 4/8-pass for the v4b-spike checkpoint might be 7/8-pass for the audit-passed checkpoint, putting it *out* of the 1–7/8 band.

The same principle applies recursively in Phase 2 and beyond — always run a fresh K=8 inference from the previous phase's best checkpoint to build the next phase's curriculum.

```bash
# K=8 T=0.9 from the audit-passed checkpoint on the full train pool.
# 16 TP=1 workers across two machines, ~60k rollouts for a 7,500-problem pool.
# Wall-clock scales with model size; reference: ~30 min on a compact (≤8B) base.
ssh <NODE_A> 'for gpu in 0 1 2 3 4 5 6 7; do
  nohup bash -c "CUDA_VISIBLE_DEVICES=$gpu <PYTHON> \
    <DATA_ROOT>/scripts/harvest.py \
      --input  <DATA_ROOT>/data/interim/<POOL>_train.jsonl \
      --output <DATA_ROOT>/data/raw/<RUN>_curriculum_K8_w${gpu}.jsonl \
      --model  <DATA_ROOT>/sft_runs/<RUN>_audit_passed_v1/merged \
      --K 8 --temperature 0.9 --top_p 0.95 --top_k 64 \
      --max_tokens 8000 --max_model_len 9024 \
      --tp 1 --gpu_mem 0.85 \
      --worker_id $gpu --n_workers 16 --batch 256 \
      --system_prompt_file <SYS_PROMPT>" > $LOG 2>&1 </dev/null &
  disown
done'
# Mirror on <NODE_B> with worker_ids 8..15.
```

### Step 7.2 — Bucket and select the 1–7/8 band (GRPO-useful curriculum)

```bash
<PYTHON> \
  <DATA_ROOT>/scripts/build_dapo_curriculum.py \
    --input_glob '<DATA_ROOT>/data/raw/<RUN>_curriculum_K8_w*.jsonl' \
    --pool <DATA_ROOT>/data/interim/<POOL>_train.jsonl \
    --out  <DATA_ROOT>/data/interim/<RUN>_dapo_curriculum_p1.jsonl \
    --lo 1 --hi 7
```

Expected K-band distribution on a competent SFT-init: roughly bimodal — a large 8/8 group, a meaningful 0/8 group, and 30–50% spread across 1–7/8. The 1–7/8 band **is** the curriculum: 0/8 contributes no within-group advantage, 8/8 contributes no within-group variance.

### Step 7.3 — DAPO config (validated; KL anchor and length penalty are load-bearing across base models)

| Parameter | Value | Notes |
|---|---|---|
| `num_generations` | 8 | GRPO group size |
| `max_completion_length` | 12,288 | 32,768 for harder pools |
| `max_prompt_length` | 1,024 | |
| `temperature` | 1.0 | Training-time only; eval at T=0 |
| `top_p` | 0.95 | |
| `learning_rate` | 1e-6 | |
| `beta` (KL) | **0.005** | Do not set to 0. Compactness collapses without the KL anchor. |
| `epsilon` (low) | 0.2 | PPO clip lower |
| `epsilon_high` | 0.28 | Decoupled clip upper |

### Step 7.4 — Reward shaping (must include all four)

1. **Three-tier accuracy** (paper Eq. for `r_acc`):
   - `1.0` if verifier accepts and post-think block is clean,
   - `0.5` if verifier accepts but post-think is verbose,
   - `0` otherwise.
2. **Soft length penalty inside `r_fmt`** — above 8,000 chars of `<think>` content, decay linearly to a maximum penalty of −0.30 at 20,000 chars: `r_fmt = r_struct − min(0.30, max(0, (|τ_think| − 8000) / 40000))`. **Soft tail, never a cliff** — cliffs cause mode collapse.
3. **Structural penalties**: counter-restart segments, sentinel-phrase rumination (≥3 "Final Answer:" markers), post-think repetition.
4. **Lenient verifier rule (v4.6.1 patch).** Lenient acceptance MUST use **post-`</think>` extraction only**. Do not scan `<think>` content for substring matches against gold — that produces false-positive 0.5 rewards on rollouts whose post-think is symbolic / non-numeric (e.g. `\cot A` for gold=1, `\frac{3√82}{11}` for gold=3). Apply `coef\sqrt{}` mask in `extract_answer`.
5. **MCQ value↔letter rescue** if the pool has multiple-choice items.

Reference implementation: `cavethought/dapo_reward.py` with v4.6.1 patches.

### Step 7.6 — Per-step rollout investigation (CRITICAL for catching drift)

After each `save_steps` boundary, fan out 8 parallel subagents — one per unique problem in the latest 64-rollout batch — to analyze reasoning logic, failure modes, notation patterns, and instability indicators. This is the `dapo-rollout-investigation` skill.

```
/dapo-rollout-investigation <DATA_ROOT>/dapo_runs/<RUN>_dapo_phase1/rollouts.jsonl <step>
```

What this catches:

- **Verbose-prose attractor.** Compact register decaying.
- **Rumination-runaway.** Median think tokens climbing past 2× baseline.
- **Counter-restart penalty-evasion.** Model segments `<think>` into 9–13 numbered passes each starting at "1." to dodge the >100-step penalty.
- **Cap-hit collapse.** >25% of rollouts hit `max_completion_length` without `</think>`.
- **Verifier false positives.** Lenient 0.5 on numerically-distinct symbolic answers.

**Stop rule.** If rollout investigation flags `severity: critical` on two consecutive checkpoints, **stop training**. The endpoint will be worse than the previous clean checkpoint. Record the failure in §15 Lessons.

### Step 7.9 — Endpoint selection from the Pareto frontier

WHETSTONE does **not** produce a single best checkpoint — it produces a Pareto frontier over accuracy and tokens-per-correct. Training past ~80% of total steps typically over-trains into rumination collapse. Document per-checkpoint accuracy and tokens-per-correct in `reports/endpoint_table.md` and pick based on downstream use:

- **ckpt-30** — token-efficiency endpoint (highest tokens-per-correct on solvable distributions).
- **ckpt-40** — slightly more capable on harder problems, but ~2× tokens.
- **ckpt-45+** — usually collapsed. Do not deploy.

## 8. End-to-End Verification

Before declaring the run complete, verify the deliverable against your evaluation suites. All evaluations use `temperature=0`, `top_p=1.0`, deterministic vLLM, and a single prompt template.

Expected qualitative outcome relative to the base model the pipeline was run on, for each suite:

The paper reports concrete numbers on MATH-500, AIME 2024 / 2025, and HMMT Feb 2025 for one base model under one training pool; treat those as a *reference shape*, not as targets your run must hit. Calibrate against (a) your base's pre-pipeline score on the same eval suite and (b) the qualitative pattern above.

If the easy-suite result fails to Pareto-improve over the base, something earlier in the pipeline regressed. Re-inspect the Stage-4 audit yield, the Stage-5 KL trajectory, and the rollout-investigation flags from Step 7.6 before deploying.

## 12. Compactness-Rigidity Tradeoff (known structural limit)

WHETSTONE compresses on the support of what the base model can already solve. On distributions where the base 8-sample pass rate is <0.20, the GRPO within-group length-ranking signal degenerates and compression collapses. Group-relative rewards can prefer shorter reasoning only when a rollout group contains multiple correct solutions; the probability of ≥2 correct rollouts in a group of size G goes roughly as `C(G,2) · p²` for small `p`, so the signal weakens quadratically at low pass rates.

Reference measurements from one published instantiation, ordered by descending base pass rate:

| Base 8-sample pass rate | Compression | Δ accuracy |
|---|---|---|
| ~0.55 | 4.2× | +16.6 |
| ~0.30 | 1.6× | +20.0 |
| ~0.20 | 1.4× | +6.6 |
| ~0.10 | 1.3× | −3.3 |

The shape — strong compression and accuracy gain at high base pass rate, marginal compression with possible accuracy regression at low base pass rate — is structural and should reproduce on any base. Use your base's per-suite 8-sample pass rate to predict where compression will be useful and where it will fail. A fix requires process-reward signal during RL (a future PRM extension); WHETSTONE as published does not include it.

## 13. Stylistic Generalization

WHETSTONE installs **compact symbolic reasoning** as one demonstration. To target a different style, change only:

1. **Stage 2 rewrite prompt** — the target register (proof-style, code-comment, Socratic, multilingual scratchpad, …).
2. **Stage 4 auditor prompt** — the per-step validity criteria for the new register.

Everything else (blind harvest, compressibility-aware SFT, cross-family audit, KL-anchored DAPO) is style-agnostic. The deterministic verifier for the underlying task remains required.

## 14. Required Environment Summary

- One GPU node with enough aggregate VRAM to host the base model in BF16 under vLLM with the chosen `max_model_len`. The validated reference is 8×80GB; smaller VRAM is supported by raising `<TP>`, shortening context, or choosing a smaller base. Optionally a second node for parallel Stage 1 harvest and Stage 5.0 curriculum K=8 (pool slices by `_uid` hash mod N).
- Python stack: Torch ≥ 2.10 (CUDA-matched), vLLM 0.19.1, flash-attn ≥ 2.8.3, FLA, causal-conv1d, and (for certain vLLM v1 / GPU combinations) the Triton patch.
- Persistent `<DATA_ROOT>` with free space ~2× the merged-checkpoint footprint per saved stage plus pool-dependent raw-harvest space (see §1.3 for the sizing rule).
- `CLOUD_API_KEY` exported on the auditor node (Stages 2.5 and 4), mapped to whatever environment variable the chosen auditor SDK reads.
- Compute time on the order of several GPU-days on the 8×80GB reference; scales roughly inversely with aggregate FLOPS.
